# Audit de code & d'architecture — 2026-07-01

> Périmètre : `apps/web`, `apps/admin`, `packages/*`, `services/*`, Docker/Caddy/CI, posture sécurité.
> Méthode : lecture ciblée (~40 fichiers dont tous les points d'entrée sensibles), balayage
> anti-patterns (grep), vérification des frontières réseau et des invariants du projet.
> Base auditée : branche `llm` (état committé). La feature « Friday booking » non pushée
> sera couverte par la **section 7** une fois le push effectué.

---

## 1. Verdict global

**Codebase de très bonne qualité, nettement au-dessus de la moyenne pour un projet solo.**

- **Architecture conforme** aux règles du projet : les navigateurs ne parlent qu'à Next.js ;
  DB, image-processor, cv-renderer et l'écriture MinIO sont strictement internes ; aucun
  composant client n'importe Prisma (vérifié par balayage).
- **SOLID réellement appliqué** : services purs injectés (`UploadPorts`, `CalendarWriter`,
  `Llm`, Prisma injecté partout), Server Actions minces qui délèguent à des services testés,
  UI décomposée en primitives partagées (`components/ui`).
- **Hygiène exemplaire** : zéro `any`, zéro `@ts-ignore`, zéro `catch` vide, zéro
  `console.log` applicatif, zéro TODO fantôme, JSDoc systématique.
- **Auth BO exemplaire** : argon2 + dummy-hash anti-timing, messages génériques
  anti-énumération, lockout compte + rate-limit IP, MFA TOTP obligatoire, sessions opaques
  hashées en DB, cookies httpOnly/secure/lax, audit `LoginAttempt`.
- **Pipeline upload conforme** à STACK_SECURITY : validation mime/taille/dimensions →
  ré-encodage webp + strip EXIF → nom randomisé → MinIO → DB.

Les constats ci-dessous sont donc des écarts ponctuels, pas un problème systémique.
Priorités : **P1** = à corriger avant la mise en ligne publique · **P2** = dette réelle à
planifier · **P3** = finition/mineur.

## 2. Constats P1 — à corriger avant la prod

### P1-1 · RBAC livré mais non câblé (DT8 connu, confirmé par l'audit)
`requirePermission` / `requireOwner` / `assertCanWrite` existent et sont testés
(`lib/auth/guards.ts`), mais **seul `ai-actions.ts` appelle `assertCanWrite`**. Les ~30 pages
du groupe `(dashboard)` reposent uniquement sur le guard du **layout**
(`app/(dashboard)/layout.tsx:11`) et toutes les mutations n'exigent que
`requireEnrolledSession` : n'importe quel rôle actif (y compris un futur VIEWER « lecture
seule ») peut tout écrire.

Il existe bien une gate edge (`apps/admin/proxy.ts` : présence du cookie de session) en
plus du guard du layout, mais elle est volontairement **coarse** (présence seulement, pas
de validation ni de rôle). Un guard **uniquement dans le layout** reste un pattern
déconseillé en App Router : le layout ne re-render pas à chaque navigation client et des
requêtes RSC partielles peuvent atteindre le segment page sans repasser par lui. La défense
en profondeur prévue par le projet (guard dans la page **et** dans l'action — le commentaire
de `guards.ts:44` le dit lui-même) n'est pas en place.

**Reco** : dérouler DT8 — `requirePermission(<module>)` en tête de chaque page **et** de
chaque Server Action, `assertCanWrite` sur toutes les mutations.

### P1-2 · CSP et Permissions-Policy absents au proxy
`proxy/Caddyfile` pose HSTS, `X-Content-Type-Options`, `X-Frame-Options`,
`Referrer-Policy` — mais **aucune `Content-Security-Policy`** ni `Permissions-Policy`,
alors que STACK_PROXY les exige. Aucune CSP côté Next non plus (vérifié).
Le site injecte du JSON-LD et rend du contenu éditable : une CSP est la dernière ligne de
défense XSS.

**Reco** : CSP par hôte (web ≠ admin ≠ stats : Umami a besoin de son script ; l'admin rend
le CV en iframe `srcdoc`) + `Permissions-Policy: camera=(), microphone=(), geolocation=()`.
À faire avant l'exposition publique.

### P1-3 · `/api/internal/*` (booking) non bloqué par Caddy sur l'hôte admin
Le Caddyfile bloque `/internal/*` (rendu CV) mais **pas `/api/internal/*`** (API booking
ajoutée depuis) : en prod, `bo.…/api/internal/appointments|availability|cancel` serait
routé vers l'app admin depuis Internet. La gate de session admin (`proxy.ts:19`) exempte
explicitement ces chemins (correct pour le machine-to-machine), donc la **seule** défense
restante est le token partagé — alors que le commentaire du code affirme « never routed by
Caddy ». Défense en profondeur cassée + création de lignes DB possible si le token fuit.
**Reco** : ajouter `handle /api/internal/* { respond 404 }` sur l'hôte admin dans le
Caddyfile (une ligne, aucun impact : le flux légitime web→admin passe par le réseau Docker
interne, pas par Caddy).

## 3. Constats P2 — dette à planifier

### P2-1 · Rate-limit public contournable (X-Forwarded-For spoofable)
Toutes les routes publiques prennent `x-forwarded-for.split(",")[0]` comme clé de
rate-limit (contact, chat, booking, cancel, appointments, availability). Caddy **ajoute**
l'IP client au header entrant sans écraser ce que le client a envoyé : le **premier**
élément est donc contrôlable par l'attaquant → contournement trivial du rate-limit
(et pollution de `LoginAttempt.ip` côté BO, qui lit aussi ce header via `parseClientIp`).

**Reco** : au proxy, écraser le header (`header_up X-Forwarded-For {client_ip}` ou poser
`X-Real-IP`) et côté app ne lire que cette valeur de confiance. Centraliser dans un helper
unique (voir P2-6).

### P2-2 · Rate-limiter en mémoire : Map jamais purgée
`packages/core/src/security/rate-limit.ts` : les buckets vides ne sont jamais supprimés de
la `Map`. Un balayage d'IPs (scan, botnet) fait croître la mémoire indéfiniment sur `web`.
**Reco** : supprimer la clé quand son tableau filtré est vide + borne max d'entrées
(éviction FIFO). Correctif de ~5 lignes, testable.

### P2-3 · Deux services de conversion d'images, dont un mort et… meilleur
- `services/converter` (Fastify + sharp) : committé, **absent du docker-compose et de la
  CI** → code mort. Ironie : c'est l'implémentation la plus durcie (limites multipart,
  `sharp.cache(false)`/`concurrency(1)` anti décompression-bomb, mime check).
- `services/image-processor` (Flask + Pillow, hérité d'OXO) : celui réellement déployé,
  mais **sans `MAX_CONTENT_LENGTH`** (borne de taille seulement chez l'appelant admin),
  gestion d'erreur qui renvoie `str(e)` au client, pas de garde décompression explicite.

**Reco** : statuer. Option A (recommandée) : basculer compose + CI sur `converter` (sharp)
et supprimer image-processor. Option B : supprimer `services/converter` et durcir Flask
(MAX_CONTENT_LENGTH, erreurs génériques, `Image.MAX_IMAGE_PIXELS` explicite, serveur WSGI).
Dans les deux cas, aligner la doc (CLAUDE.md parle de « converter », le compose de
« image-processor »).

### P2-4 · i18n web : deux systèmes, dont un quasi mort
next-intl est câblé (middleware, `messages/fr|en.json` chargés à chaque requête), mais
**aucun composant ne consomme les catalogues** (seul `useLocale` est utilisé). Les libellés
sont en dur dans les composants : objets `LABELS` (`public-cv.tsx`), ternaires
`isEn ? … : …` (`[locale]/layout.tsx:78-82`). Double source de vérité, catalogues morts.
**Reco** : choisir — soit tout basculer sur les messages next-intl, soit assumer les
LABELS colocalisés et supprimer les catalogues. L'état hybride est le pire des deux.

### P2-5 · `estimateTokens` dupliqué avec deux formules divergentes
`apps/web/app/api/chat/route.ts:9` (`len/4`) vs `apps/admin/lib/actions/ai-actions.ts:12`
(`len*2/4`). Même concept, deux résultats : le budget mensuel n'est pas décompté de façon
cohérente entre le chatbot public et l'assistance BO.
**Reco** : une seule implémentation dans `packages/core/src/ai/budget.ts` (qui possède déjà
`assertBudget`/`recordUsage`), paramètre explicite si un facteur de marge est voulu.

### P2-6 · DRY : boilerplate dupliqué dans les routes publiques et les actions
- `clientIp()` copié-collé dans **6 route handlers** web + variante `parseClientIp` admin.
- Séquence « rate-limit → parse JSON → honeypot → Zod » dupliquée (contact, booking…).
- Helpers `str()`/`csv()`/`lines()` recopiés dans `content-actions.ts`, `cv-actions.ts`,
  `crm-actions.ts` (+ `reqId`).
**Reco** : `apps/web/lib/http/public-route.ts` (ip de confiance + garde commune) et
`apps/admin/lib/actions/form-utils.ts`. Se combine avec P2-1.

### P2-7 · `content-actions.ts` : god file multi-domaines (360 l.)
KPI + profil + CV HTML + sections home + skills + FAQ + réglages + carrière + analyses
dans un seul fichier d'actions. Chaque action est propre, mais le fichier viole SRP et va
continuer à grossir.
**Reco** : scinder par domaine (`kpi-actions.ts`, `career-actions.ts`, `analysis-actions.ts`…),
comme c'est déjà fait pour articles/projets/media.

## 5. Architecture — revue détaillée

**Conforme et saine.** Points vérifiés :

| Invariant | État |
|---|---|
| Seul le proxy expose un port (prod) | ✔ (modulo P3-8 : ports dev dans le fichier de base) |
| DB/converter/écriture MinIO sur réseau `internal` sans Internet | ✔ |
| Prisma uniquement côté serveur | ✔ (balayage : aucun composant client ne l'importe) |
| Moindre privilège DB (`app_web` RO / `app_admin` RW / `umami` isolé) | ✔ (+ REVOKE sur les nouveaux modèles) |
| Zod aux frontières | ✔ routes publiques & services contenu ; ✖ 3 exceptions (P3-6) |
| Web → RDV privés via route interne token-guardée (jamais la DB en direct) | ✔ (`admin-client.ts`, fail-closed) |
| Conteneurs non-root, multi-stage, standalone | ✔ (web/admin `USER node`, cv-renderer read-only + no-new-privileges) |
| Secrets hors code/image (`.dockerignore` exclut `.env*`) | ✔ |
| CI : lint+typecheck+test+build & E2E requis, images taguées sur release only | ✔ |
| Doc vs réel | ✖ écarts : CLAUDE.md dit « converter » (réel : image-processor), 7 services annoncés (réel : 9 avec migrate/minio-init/cv-renderer), `packages/ui` absent du schéma d'archi |

**Frontière web→admin (booking)** : bonne conception — le web (rôle RO) ne touche jamais
aux tables RDV en écriture, il passe par les routes internes de l'admin avec token partagé,
fail-closed si le token n'est pas configuré, `SlotTakenError` mappée sur 409, index unique
partiel comme garde anti double-réservation. Rien à redire sur le design.

**Qualité des tests** : réelle (services purs testés, E2E sur les parcours critiques,
politique de throttle testée à part). Non re-exécutés dans cet audit (une autre session
travaille dans l'arbre) — la CI fait foi sur les commits.

## 6. Critique produit / UX (passe navigateur desktop 1280 + mobile 390 effectuée)

Vérifié en navigateur réel (MCP Playwright) : home (scroll-reveal OK, DA respectée,
sections riches), `/cv` mobile (rendu premium, dates alignées, PDF FR/EN), `/contact`,
`/temoignages`, chat + formulaire RDV desktop et mobile, BO mobile (dashboard, RDV).

**Constats UX (par impact décroissant) :**

1. **Deux systèmes de RDV incohérents pour le visiteur** : `/contact` garde l'ancien
   formulaire « Demander un rendez-vous » avec **créneau libre** (`datetime-local`,
   n'importe quelle date/heure), pendant que le chat Friday propose les **vraies
   disponibilités**. Un RDV « contact » pris sur un créneau occupé/indisponible passe
   (il ne bloque pas de créneau et sa confirmation contourne la logique de dispo).
   → Unifier : brancher le bloc RDV de `/contact` sur le sélecteur de créneaux réels
   (ou le reformuler en « proposer une plage » clairement distincte).
2. **404 brute** : aucun `not-found.tsx` / `error.tsx` / `global-error.tsx` dans
   `apps/web` → « 404: This page could not be found. » par défaut de Next sur un slug
   inexistant (vérifié). Pour une vitrine, une 404 à la DA avec lien retour est due.
3. **Sélecteur de créneaux : 132 options dans un `<select>` plat** (2 semaines × ~10
   créneaux/jour, vérifié). Utilisable mais pénible, surtout au desktop.
   → Regrouper par jour (`<optgroup>`) a minima ; idéalement un choix jour → heure.
   Ajouter la **durée (30 min)** et le fuseau dans le libellé du formulaire.
4. **Chat/booking FR-only sur un site bilingue** : accueil de Friday, libellés du
   formulaire, `slotLabel` et messages d'état sont codés en dur en français — un visiteur
   sur `/en` reste servi en français (rejoint P2-4).
5. **BO `/rdv` : pas d'onglet « Annulés »** alors que le statut CANCELLED existe depuis le
   self-service cancel (onglets vérifiés : Tous / En attente / Confirmés / Refusés).
6. **Bulle de chat flottante** : en mobile elle peut recouvrir des champs de formulaire en
   bas de page (vu sur `/temoignages`) — vérifier qu'elle ne masque jamais un bouton
   d'envoi ; prévoir un décalage quand un formulaire a le focus si besoin.
7. **Feedback d'erreur serveur des formulaires publics** : le formulaire booking gère
   explicitement 409/erreur (bien). À harmoniser sur contact/témoignages (message visible
   pour 400/429/502, pas d'échec muet) — l'upload BO retourne silencieusement sur fichier
   invalide (`media-actions.ts:15`).
8. **Accessibilité** : bonne base (aria-labels, `aria-expanded`/`aria-pressed`, kanban
   clavier). Manquent : `aria-live` sur le fil du chat (les réponses ne sont pas annoncées
   aux lecteurs d'écran), skip-link, et un indicateur « Friday écrit… » pendant l'attente
   (le bouton se désactive mais rien dans le fil).
9. **BO Médias : aucune suppression possible** (vérifié : le panneau de détails n'offre
   qu'« Ouvrir l'original ») — un média uploadé par erreur reste pour toujours (DB + MinIO).
   Prévoir suppression avec garde « média utilisé par X contenus ».

## 7. Feature « Friday booking » — audit de l'état final pushé

**Verdict : bien conçue et bien exécutée.** Vérifié sur les 11 commits pushés + parcours
navigateur réel (formulaire dans le chat, créneaux réels chargés, desktop + mobile).

Points forts : formulaire dédié **sans extraction de champs par le LLM** (guardrails mis à
jour pour interdire l'invention de créneaux), double garde anti double-réservation
(re-check dispo + index unique partiel → 409), token d'annulation aléatoire + réponse
générique anti-énumération, emails lifecycle best-effort qui ne bloquent jamais la
réservation, affichage Europe/Paris corrigé côté BO.

Écarts relevés (repris dans les sections ci-dessus) : P1-3 (Caddy ne bloque pas
`/api/internal/*`), §6.1 (coexistence avec le RDV libre de `/contact`), §6.3 (select plat,
durée non affichée), §6.4 (FR-only), §6.5 (onglet Annulés BO), P2-6 (boilerplate
`clientIp`/honeypot dupliqué dans les nouvelles routes aussi). Reste ouvert côté arbre de
travail : la modif non commitée du Dockerfile image-processor (`ENV HOME=/app`) et de
`.gitignore` (ignore de `docker-compose.override.yml`) — à assumer ou jeter.
